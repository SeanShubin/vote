# Invite-Link Authentication (Chain of Trust)

An alternative authentication model that eliminates passwords and email verification. Trust is established through a chain of vouching: the system owner trusts admins, admins trust the people they invite.

## Concept

Instead of registering with a password, users receive a one-time invite link from an admin. Claiming the invite produces a personal secret link that acts as the user's credential. The admin never learns the secret link, so they cannot impersonate the user they invited — which matters in a voting system where ballot integrity depends on voter identity.

## Flow

### Admin creates an invite
1. Admin requests a new invite link in the UI
2. System generates a short-lived, one-time-use URL (e.g. `/invite/a3f8c2d9...`)
3. Admin delivers the URL to the intended recipient via any out-of-band channel (chat, in person, etc.)
4. The invite expires if unused after a configurable window (e.g. 48 hours)

### New user claims the invite
1. User visits the invite URL
2. User enters only a username — no password, no email
3. System creates the account, marks the invite as consumed, and generates a long random secret token
4. System displays the user's personal secret URL (e.g. `/me/s9k2p7f1...`) with instructions to bookmark it
5. The secret URL is shown exactly once; the system stores only a hash of it

### Ongoing access
- Visiting the secret URL auto-authenticates the user with no login form
- The user can request a new secret URL at any time, which invalidates the previous one
- An admin can revoke access by deactivating the account; a new invite can be issued to restore it

## Security Properties

| Property | Analysis |
|---|---|
| Admin impersonation | Blocked — the invite is one-time-use and the secret URL is generated after it is consumed, so the admin never has it |
| Ballot secrecy | Preserved — an admin cannot cast a vote as an invited user because they do not know the user's secret URL |
| Secret URL exposure | Risk — the URL can appear in browser history, bookmarks, or server logs; HTTPS is mandatory and the URL should not be logged |
| Link theft | If the secret URL is observed or guessed, the holder can authenticate as the user; mitigated by using a cryptographically random token of sufficient length |
| Account recovery | No self-service recovery; a locked-out user requires the admin to revoke the old account and issue a fresh invite |
| Invite forwarding | An invite link can be claimed by anyone who receives it before the intended recipient; admins should treat invite URLs as sensitive until claimed |

## Tradeoffs vs. Password Authentication

**Advantages**
- No email infrastructure required
- No password reset flows or credential storage
- Simpler user experience — bookmark once, click to access
- Natural chain of trust: every account traces back to a human vouching for the user
- Admin cannot impersonate voters

**Disadvantages**
- A lost or compromised secret URL cannot be self-recovered — the user needs admin intervention
- The secret URL is a plaintext credential that lives in the browser; it is only as safe as the device
- No support for multi-factor authentication in the basic model
- Sharing a device means sharing access unless the user regenerates their link on each device

## Proposed Requirement Changes

Replacing the password-based account and access-control requirements with:

- Generate a one-time invite link with a configurable expiration time (admin)
- Deliver an invite link to a prospective user through any trusted channel
- Revoke an unused invite link before it is claimed
- Claim an invite by choosing a username, with no password required
- Receive a personal secret link upon claiming an invite, which the issuing admin cannot see
- Bookmark the personal secret link to log in automatically on return visits
- Request a new personal secret link at any time, which invalidates the previous one
- Lose access if the personal secret link is lost until an admin issues a new invite
- Have an admin revoke access by deactivating an account

## Implementation Sketch

Two token types are needed:

**Invite token** — short-lived, single-use
- Generated by an admin action
- Stored with: `createdBy`, `expiresAt`, `claimedBy` (null until used), `claimedAt`
- Delivered as a URL parameter; invalidated on first use or expiry

**Secret access token** — long-lived, rotatable
- Generated when an invite is claimed
- Only a hash (e.g. SHA-256) stored server-side; the plaintext is shown to the user once
- Presented as a URL path segment on every request; looked up by hash to resolve the user session
- Rotated by generating a new token, storing its hash, and discarding the old hash

This fits naturally into the existing event-sourcing model:
- `InviteGenerated` — records invite creation
- `InviteClaimed` — records username chosen and invite consumed
- `AccessTokenRotated` — records that the old token hash was replaced

No passwords, no email addresses, no session cookies required.
